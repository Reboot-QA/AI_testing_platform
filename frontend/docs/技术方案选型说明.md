# 技术方案选型说明（前端）

> 目的：记录「是否把前端从 Vue + JS 重构为 React + TS」这一决策的评估过程、结论与前提，避免日后反复纠结方向
> 结论先行：**留在 Vue 3 + TypeScript 渐进增强，不迁 React**
> 版本：v1.0（2026-07）

---

## 1. 背景

平台前端原为 **Vue 3 + JavaScript**（约 2.1 万行 SFC、75+ 个 `.vue`、100% `<script setup>`），工程化基本为零（无 ESLint/Prettier/TS/测试/Git 钩子）。随着功能增多、要对接其他系统，需评估两条路线：

- **A. 留在 Vue 3 + TS**（渐进增强，补齐工程化 + 双轨 UI）
- **B. 重构为 React + TS**

## 2. 对比维度

| 维度 | React + TS | Vue 3 + TS | 说明 |
| --- | --- | --- | --- |
| **AI 生态/语料** | ✅ 强（语料约 4–5×，v0/shadcn、AI 聊天组件 React 优先） | ⚠ 弱一些 | React 侧 AI 生成 UI 可用率更高 |
| **重写成本** | ❌ 2.1 万行全量重写，功能停滞 4–8 周 | ✅ 无需重写，边跑边补 | 最大的成本项 |
| **回归风险** | ❌ 高（Monaco、拖拽编排、SSE 实时视图都要重写） | ✅ 低 | apifox 新模块刚用 Vue 重构完且质量好 |
| **对接其他系统** | 平手 | 平手 | 以 iframe / API 形式，框架无关；且周边系统多为 Vue |
| **前端性能** | 平手 | 平手 | Vue 3.6 Vapor 在途；两端都够用 |
| **数据缓存** | 平手（TanStack Query） | 平手（TanStack Query 亦可用） | 方案双端通用 |
| **人才/维护** | 需要 React 经验 | 团队已在 Vue | 沿用降低门槛 |

## 3. 决策与理由

**选 A：留在 Vue 3 + TS 渐进增强。** 关键权衡：

- React 唯一的实质优势是 **AI 生态/生成 UI**，但被「**2.1 万行全量重写 + 4–8 周功能停滞 + 高回归风险**」抵消；
- apifox 新模块（当前开发重心）刚用 Vue 完成高质量分层重构，重写等于推倒重来；
- 对接其他系统是 iframe/API 形式，**框架无关**，且周边系统以 Vue 为主；
- 性能与缓存两端基本平手。

## 4. 如何弥补 Vue 在 AI 生态上的差距

把「React 的 AI 优势」用工程手段补回来，已纳入落地：

1. **规范喂 AI**：产出 [《前端开发规范》](./前端开发规范.md) 与 [《前端 UI 设计规范》](./前端UI设计规范.md)，并在根 `CLAUDE.md` 内联关键规则——让 AI 辅助开发自动遵循团队约定。
2. **ESLint/Prettier 固化风格**：flat config + 提交钩子，把「怎么写才对」变成可自动执行的规则，降低 AI 产出的返工。
3. **引入 Tailwind v4 + shadcn-vue**：获得接近 React 的「AI 生成 UI / 复制即用组件」能力，同时不破坏 Element Plus 存量。
4. **TypeScript + 后端类型自动生成**：`Schemas['XxxOut']` 让 AI 生成代码有确定类型约束，少幻觉、少踩坑。

## 5. 何时值得重新评估（前提条件）

出现以下情况之一，可重启 React 评估：

- 周边协作系统整体转向 React，Vue 的「生态一致性」优势消失；
- 「AI 整页生成 UI」成为核心工作流，Vue 侧生成可用率成为明显瓶颈；
- 团队人才结构转向 React 为主。

在此之前，**Vue 3 + TS 渐进增强是成本、风险、收益最优解**。
