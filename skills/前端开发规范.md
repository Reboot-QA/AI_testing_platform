## 一、技术栈与工具约定

| 类别 | 推荐方案 |
|------|----------|
| 框架 | Vue 3（Composition API 优先） |
| 构建工具 | Vite |
| 语言 | TypeScript |
| 状态管理 | Pinia |
| 路由 | Vue Router |
| UI 组件库 | Element Plus / Ant Design Vue（任选即可） |
| 请求库 | Axios |
| 代码校验 | ESLint + Prettier |
| 提交校验 | husky + lint-staged + commitlint |

工具配置以自动化为主，尽量减少依赖人工遵守的约定。

## 二、代码风格规范

### 2.1 基本约定

- 缩进：**2 个空格**，禁止使用 Tab。
- 字符串：统一使用**单引号** `'`，模板字符串除外。
- 分号：语句末尾**是否加分号团队内保持统一**（推荐加）。
- 每行代码长度建议不超过 **100 字符**。
- 文件末尾保留一个空行。

### 2.2 工具配置

- 使用 **ESLint** 检查代码质量，**Prettier** 负责格式化。
- 通过 `eslint-config-prettier` 关闭两者冲突的规则。
- 提交前用 **husky + lint-staged** 做 pre-commit 校验，不合规范无法提交。

```bash
# 安装示例
npm i -D eslint prettier eslint-config-prettier husky lint-staged
```

### 2.3 编码习惯

- 优先使用 `const`，需要重新赋值时才用 `let`，禁止使用 `var`。
- 优先使用箭头函数，注意 `this` 指向。
- 使用可选链 `?.` 和空值合并 `??` 简化判空。
- 避免魔法数字/字符串，抽取为常量。

## 三、命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 变量 / 函数 | 小驼峰 camelCase | `userName`、`getUserInfo` |
| 常量 | 大写下划线 | `MAX_COUNT`、`API_BASE_URL` |
| 组件文件 | 大驼峰 PascalCase | `UserCard.vue` |
| 组件名（模板中） | 大驼峰或短横线 | `<UserCard>` / `<user-card>` |
| 普通 JS/TS 文件 | 小驼峰或短横线 | `formatDate.ts`、`user-service.ts` |
| CSS 类名 | 短横线 / BEM | `user-card__title--active` |
| 布尔值 | is / has / can 前缀 | `isLoading`、`hasPermission` |
| 事件方法 | handle / on 前缀 | `handleClick`、`onSubmit` |
| 类型 / 接口（TS） | 大驼峰 | `interface UserInfo` |

命名要**语义化**，避免 `data1`、`temp`、`a` 等无意义命名。

## 四、目录结构规范

```
src/
├── api/            # 接口请求，按模块拆分
│   ├── user.ts
│   └── order.ts
├── assets/         # 静态资源（图片、字体等）
├── components/     # 全局通用组件
├── views/          # 页面级组件
├── router/         # 路由配置
├── store/          # 状态管理（Pinia）
├── composables/    # 可复用组合式逻辑（use* hooks）
├── utils/          # 工具函数
├── directives/     # 自定义指令
├── styles/         # 全局样式、变量、mixin
├── constants/      # 常量、枚举
├── types/          # TS 类型定义
└── App.vue
```

原则：**按功能/职责组织**，而非按文件类型堆放，便于扩展和查找。

## 五、Vue 组件规范

### 5.1 组件基本约定

- **优先使用 `<script setup>` + Composition API**。
- 单个组件建议不超过 **300 行**，超出应拆分子组件或抽离逻辑。
- 单文件组件（SFC）标签顺序统一为：`<script>` → `<template>` → `<style>`（或团队统一其他顺序）。

```vue
<script setup lang="ts">
import { ref, computed } from 'vue'

interface Props {
  title: string
  count?: number
}

const props = withDefaults(defineProps<Props>(), {
  count: 0,
})

const emit = defineEmits<{
  (e: 'update', value: number): void
}>()

const isActive = ref(false)
const doubled = computed(() => props.count * 2)

function handleClick() {
  emit('update', doubled.value)
}
</script>

<template>
  <div class="user-card" :class="{ 'is-active': isActive }">
    <h3 class="user-card__title">{{ title }}</h3>
    <button @click="handleClick">点击</button>
  </div>
</template>

<style scoped>
.user-card__title {
  font-size: 16px;
}
</style>
```

### 5.2 Props 规范

- Props 必须**明确类型**，可选项提供**默认值**。
- 命名使用小驼峰，模板中使用短横线。
- 避免直接修改 props，需变更时通过 emit 通知父组件。

### 5.3 事件规范

- 事件名使用短横线（kebab-case）：`@update-user`。
- 方法命名使用 `handle` / `on` 前缀。

### 5.4 其他约定

- `v-for` 必须绑定唯一 `:key`，禁止使用 index 作为 key（列表会变动时）。
- `v-if` 与 `v-for` **不要用在同一元素**上。
- 组件内复用逻辑抽成 `composables/useXxx.ts`。
- 谨慎使用 `v-html`，防止 XSS。

## 六、样式规范

- 组件样式统一加 **`scoped`** 或使用 **CSS Modules**，避免全局污染。
- 全局变量（主题色、间距、字号）统一定义在 `styles/variables.scss` 或 CSS 自定义属性中，**禁止硬编码颜色值**。
- 类名推荐使用 **BEM** 命名法：`block__element--modifier`。
- 移动端适配方案（rem / vw / postcss-px-to-viewport）团队统一。
- 避免深层选择器嵌套，建议不超过 **3 层**。

```scss
// styles/variables.scss
:root {
  --color-primary: #409eff;
  --spacing-md: 16px;
}
```

## 七、API 与请求规范

- 请求统一封装 Axios 实例，集中处理 baseURL、超时、请求/响应拦截、错误提示、token 注入。
- 接口按业务模块拆分到 `api/` 目录，页面组件不直接写请求 URL。
- 统一处理 loading、错误码、鉴权失效跳转。

```typescript
// api/request.ts
import axios from 'axios'

const service = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 10000,
})

service.interceptors.request.use((config) => {
  const token = localStorage.getItem('token')
  if (token) config.headers.Authorization = `Bearer ${token}`
  return config
})

service.interceptors.response.use(
  (res) => res.data,
  (error) => {
    // 统一错误处理
    return Promise.reject(error)
  },
)

export default service
```

## 八、状态管理规范

- 使用 **Pinia** 管理全局状态。
- 按业务模块拆分 store，命名 `useXxxStore`。
- 只把**跨组件共享**的状态放入 store，组件内部状态用 `ref` / `reactive`。
- 避免在 store 中放置过多临时/派生数据，派生数据用 `computed` / getters。

```typescript
// store/user.ts
import { defineStore } from 'pinia'

export const useUserStore = defineStore('user', {
  state: () => ({
    userInfo: null as UserInfo | null,
  }),
  getters: {
    isLogin: (state) => !!state.userInfo,
  },
  actions: {
    setUser(info: UserInfo) {
      this.userInfo = info
    },
  },
})
```

## 九、Git 提交规范

采用 **Conventional Commits（Angular 规范）**，格式：

```
<type>(<scope>): <subject>
```

### 常用 type

| type | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | 修复 bug |
| `docs` | 文档变更 |
| `style` | 代码格式（不影响逻辑） |
| `refactor` | 重构 |
| `perf` | 性能优化 |
| `test` | 测试相关 |
| `chore` | 构建、依赖、工具等杂项 |
| `revert` | 回滚 |

**示例**：`feat(login): 增加短信验证码登录`

### 分支命名

- `feature/xxx`：新功能
- `bugfix/xxx`：修复缺陷
- `hotfix/xxx`：线上紧急修复
- 禁止直接向 `main` / `master` / `develop` 提交，需通过 PR/MR 合并。

建议用 **commitlint** 强制校验提交信息格式。

## 十、注释与文档规范

- 复杂业务逻辑、特殊处理、临时方案必须写注释。
- 注释解释**「为什么这么做」**，而非复述代码「做了什么」。
- 待办与问题统一标记：`// TODO:` 待办、`// FIXME:` 待修复。
- 公共函数、组件建议用 **JSDoc** 说明参数与返回值。
- 每个可复用组件建议在头部注释说明用途、Props、Events。

```typescript
/**
 * 格式化金额，保留两位小数并添加千分位
 * @param amount 金额数值
 * @returns 格式化后的字符串，如 "1,234.56"
 */
function formatMoney(amount: number): string {
  return amount.toLocaleString('zh-CN', { minimumFractionDigits: 2 })
}
```

## 十一、性能与优化建议

- 路由与大组件使用**懒加载**（动态 import）。
- 长列表使用**虚拟滚动**（如 vue-virtual-scroller）。
- 合理使用 `v-once`、`v-memo` 缓存静态内容。
- 图片资源压缩，使用 WebP，配合懒加载。
- 生产构建开启代码分割、Tree Shaking、gzip/brotli 压缩。
- 避免在模板中写复杂表达式，改用 `computed`。
- 及时销毁定时器、事件监听，防止内存泄漏。
