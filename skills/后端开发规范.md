## 二、分层架构规范

**单向依赖，禁止跨层调用**：

```
router  →  service  →  repository  →  model / DB
（校验编排）  （业务逻辑）   （数据访问）      （ORM 实体）
```

| 层 | 只做 | 禁止 |
|----|------|------|
| **router** | 参数校验、权限/归属校验、编排 service、异常转 HTTP | 写业务逻辑、直接查/写 DB |
| **service** | 业务逻辑、事务边界（`commit`）、跨 repo 编排 | 拼装 HTTP 响应、感知 `Request`/状态码 |
| **repository** | 单表/子表 CRUD、查询构造 | 业务校验、权限、`commit`、抛业务异常 |
| **model** | ORM 实体，承载数据与不变量 | 塞业务方法 |

- **v2 / 新模块**：严格三层落地，数据访问抽到 repository。正面范式见 `app/{routers,services,repositories,models}/apifox/`。
- **存量**：service 直接访问 DB（无 repository 层），**不倒查、不强制回改**；但新增代码不得沿用旧模式。
- 反面教材（禁止再出现同类）：`routers/api_automation.py`(1918 行) —— router 里塞满业务与 DB 访问。

### router 典型骨架（归属校验前置）

```python
router = APIRouter(prefix="/apifox", tags=["接口自动化v2·用例"])

def _case_checked(db: Session, cid: int, user: User) -> ApifoxEndpointCase:
    case = case_repo.get_case(db, cid)
    if not case:
        raise HTTPException(status_code=404, detail="用例不存在")
    get_accessible_project(db, case.project_id, user)  # 权限/归属
    return case

@router.put("/cases/{cid}", response_model=CaseOut)
def update_case(cid: int, payload: CaseUpdate,
                db: Session = Depends(get_db),
                user: User = Depends(get_current_user)) -> CaseOut:
    _case_checked(db, cid, user)
    try:
        return service.update_case(db, cid, payload)
    except ConflictError as e:
        raise HTTPException(status_code=409, detail=e.message)
```

## 三、目录结构规范

```
backend/app/
├── main.py            # FastAPI 应用装配、路由注册、生命周期
├── config.py          # pydantic-settings 配置
├── database.py        # engine / SessionLocal / Base / get_db
├── auth.py            # 鉴权依赖（get_current_user）
├── request_logging.py # 请求日志中间件（request_id）
├── bootstrap.py       # 启动初始化
├── schemas.py         # 存量全局 schema（新模块就近放模块目录）
├── routers/           # 路由层，按业务模块拆分
│   └── apifox/        # v2 模块：路由 + 就近的 *_schemas.py
├── services/          # 业务层
│   └── apifox/        # v2 业务 + errors.py（模块内业务异常）
├── repositories/      # 数据访问层（v2）
│   └── apifox/        # *_repo.py，一表一文件
├── models/            # ORM 实体
│   └── apifox/        # 按聚合拆文件
├── constants/         # 常量、枚举
└── utils/             # 纯函数工具（禁止 common/misc 兜底命名）
```

原则：**按业务模块/职责组织**，同一功能的 router/service/repo/model/schema 用一致的模块名（如 `apifox/case*`）串起来，便于查找。新功能优先在对应分层**新建文件**，不往已有大文件里堆。

## 四、代码风格规范

- 缩进 **4 空格**，遵循 PEP 8，由 ruff 统一格式化。
- 所有**公开函数完整类型注解**；可空返回写 `X | None`（不用 `Optional[X]` 新写法二选一，项目内保持统一——现状 model 用 `Optional[...]`，函数签名用 `X | None`）。
- 优先早返回（guard clause）减少嵌套，禁止深层 `if` 金字塔。
- 避免魔法数字/字符串，枚举值抽到 `constants/` 或模块级常量（如 `CASE_CATEGORIES`）。
- 禁止 `print`，一律用 stdlib logging（见 [日志规范](#十一日志规范)）。
- import 顺序：标准库 → 第三方 → 本项目，组间空行（ruff 自动整理）。

## 五、命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 变量 / 函数 | 小写下划线 snake_case | `project_id`、`get_case` |
| 常量 / 枚举值 | 大写下划线 | `MAX_RETRY`、`CASE_CATEGORIES` |
| 类 / Pydantic / ORM | 大驼峰 PascalCase | `CaseUpdate`、`ApifoxEndpointCase` |
| 模块文件 | 小写下划线 | `case_service.py`、`case_repo.py` |

## 六、Pydantic 与数据契约

- **层间业务数据用 pydantic model / dataclass 承载，禁止裸 dict**（反面即存量 `__meta` blob）。
- 请求体 / 响应体 schema 与业务实体分离：`XxxCreate`、`XxxUpdate`、`XxxOut`。
- schema 就近放模块（v2 放 `routers/apifox/*_schemas.py`），与前端 `frontend/src/api/` 契约对齐。
- 结构化 JSON 列用 pydantic model 做 dump/load，坏数据兜底为空对象而非崩溃：

```python
def _load_request_spec(text: str | None) -> RequestSpec:
    if not text:
        return RequestSpec()
    try:
        return RequestSpec.model_validate_json(text)
    except ValueError:
        return RequestSpec()  # 脏数据不阻断读取
```

- 对象**构造即有效**：创建后即处于合法状态，不允许"半初始化再补字段"。

## 七、数据库设计规范

### 7.1 表与列

- 表名：**复数 + 小写下划线 + 模块前缀**，如 `apifox_endpoint_cases`、`apifox_case_assertions`。
- 主键统一 `id: Mapped[int]`，`primary_key=True, index=True`。
- 外键显式声明 `ForeignKey("projects.id")`，并对查询列加 `index=True`。
- 时间戳统一 `created_at` / `updated_at`：

```python
created_at: Mapped[datetime] = mapped_column(DateTime, default=datetime.utcnow)
updated_at: Mapped[datetime] = mapped_column(
    DateTime, default=datetime.utcnow, onupdate=datetime.utcnow
)
```

- 排序用 `sort_order: Mapped[int]`，配 `id` 做稳定次序。
- 枚举类字段用短 `String(20)` + 注释列出取值，不用数据库 enum（迁移成本高）；取值集中在代码常量。

### 7.2 SQLAlchemy 2.0 写法

- 一律用 `Mapped[...]` + `mapped_column(...)` 声明式（禁止旧式 `Column(...)`）。
- 可空列 `Mapped[Optional[str]] = mapped_column(Text, nullable=True)`。
- 需要 DB 默认值时同时给 `default=` 与 `server_default=`（兼容已存在行）：

```python
category: Mapped[str] = mapped_column(String(20), default="other", server_default="other")
```

### 7.3 结构化优先，拒绝大 JSON blob

- **可查询 / 可报表的字段拆成行级子表**，不塞进一个大 JSON。
  - 正面：断言/提取拆 `apifox_case_assertions` / `apifox_case_extracts` 子表——这是拆 `__meta` blob 的核心动机。
  - 仅"整体读写、无需单字段查询"的配置（如 `request_spec`）才用结构化 JSON 列（`Text` + pydantic 承载）。

### 7.4 并发与一致性

- **共享可变状态禁止"读改写整块 JSON"**（并发下 last-write-wins 丢更新）：用行级表 + 针对性 `UPDATE`。
- 多人协同编辑的聚合根加**乐观锁 `version` 列**，保存时校验期望版本，不一致抛 `ConflictError`：

```python
version: Mapped[int] = mapped_column(Integer, default=1, server_default="1")
```

- 子表"传全量则先删后插"（bulk replace）由 service 编排，避免并发覆盖。

### 7.5 迁移

- 表结构变更配套迁移脚本；`default` + `server_default` 保证旧行有值。
- 不在 `docker/`、`deploy.sh`、`.github/` 等部署文件里塞 DDL，除非任务明确要求。

## 八、数据访问（Repository）规范

- **一个 repo 文件对应一个聚合/实体**（`case_repo.py`、`endpoint_repo.py`）。
- repo 函数只做查询/CRUD，**不含业务校验、不含权限、不 `commit`**（事务边界在 service）。
- 函数签名首参恒为 `db: Session`，返回 ORM 实体或 `Row` 元组：

```python
def get_case(db: Session, case_id: int) -> ApifoxEndpointCase | None:
    return db.query(ApifoxEndpointCase).filter(ApifoxEndpointCase.id == case_id).first()

def list_cases(db: Session, endpoint_id: int) -> list[ApifoxEndpointCase]:
    return (
        db.query(ApifoxEndpointCase)
        .filter(ApifoxEndpointCase.endpoint_id == endpoint_id)
        .order_by(ApifoxEndpointCase.sort_order, ApifoxEndpointCase.id)
        .all()
    )
```

- 查询列表默认给稳定排序（`sort_order, id`），避免分页/前端次序抖动。
- 同一查询逻辑出现 **2 次即抽取**到 repo 函数，不在 service 里复制粘贴 query。

## 九、异常与错误处理

- **业务错误抛自定义异常，禁止 `except Exception: pass`**。
- **v2 / 新模块**业务异常放模块内 `errors.py`，router 统一转 HTTP 状态码：

| 异常 | HTTP | 场景 |
|------|------|------|
| `ConflictError` | 409 | 乐观锁版本冲突 |
| `ValueError` | 400 | 参数/业务校验失败 |
| 资源不存在 | 404 | router 内 `HTTPException(404)` |

```python
class ConflictError(Exception):
    """乐观锁保存冲突，携带当前版本供前端重载。"""
    def __init__(self, current_version: int, message: str = "内容已被他人修改，请刷新后重试"):
        self.current_version = current_version
        self.message = message
        super().__init__(message)
```

- service 抛领域异常（`ConflictError`/`ValueError`），**只有 router 感知 HTTP**（`HTTPException`）——service 层不 import `fastapi`。
- 捕获异常必须处理或重抛并保留上下文，不吞掉。

## 十、异步与并发

> 执行引擎、定时调度、AI 批量生成是本平台并发热点，以下几条对应评审真实踩过的坑。

- **async 端点不阻塞**：`async def` 内禁止 `time.sleep` / `requests` / 同步 DB / CPU 密集循环（卡 event loop）。要阻塞就写同步 `def` 端点，或 `asyncio.to_thread` 下放；HTTP 出站统一 `httpx.AsyncClient`。
- **长任务不占请求线程**：套件/场景执行、批量生成用后台任务 / SSE 推进，请求端建任务即返回，且能超时中断。
- **状态流程必须 `try/finally` 兜底**："先置 `running` → 干活 → 写终态"，中间必须兜底写失败态，避免进程挂掉永久卡 `running`。
- **DB session 不共享**：每请求/任务/后台线程/SSE 生成器用**独立 session** 自管生命周期；批量写单次 `commit`，不在循环里逐条 commit。
- **并发扇出限流**：信号量 + `as_completed` 流式、单批失败隔离（正面范式 `services/ai_service.py`）。
- **后台线程/调度**：沿用单例守护线程 + 轮询（`schedule_service`），到期扫描防重入，轮询循环内异常 `try/except` 兜住，勿因一条坏数据打死整个线程；线程内用独立 session。

详见 `.claude/rules/concurrency.md`。

## 十一、日志规范

- 用 **stdlib logging**，禁止 `print`。
- 复用现有 `request_logging` 的 `request_id`，便于串联一次请求的全链路日志。

```python
import logging
logger = logging.getLogger(__name__)

logger.info("用例保存成功 case_id=%s version=%s", case_id, version)
logger.warning("请求体解析失败，兜底为空 spec case_id=%s", case_id)
```

- 用 `%s` 惰性格式化，不用 f-string 拼接日志（避免无谓字符串构造）。
- 不记录敏感信息（密码、token 明文）。

## 十二、文件规模与拆分（硬性，防"上帝文件"）

| 单位 | 软上限 | 硬上限（必须拆） |
|------|--------|------------------|
| Python 单文件 | ≤ 400 行 | > 600 行 |
| 单函数 | ≤ 50 行 | — |
| 单类 | ≤ 300 行 | — |

- 反面教材：`routers/api_automation.py`(1918 行)。禁止再出现同类。
- 拆分手段：按职责拆 service / repository；纯逻辑抽 `utils/` 纯函数（正面范式 `utils/apiCaseConfig.js` 的后端对应物）。
- **高内聚低耦合**：一个模块只做一件事，对外暴露最小接口；跨模块通过明确接口协作，不直接读写对方内部结构；禁止循环依赖。

## 十三、测试规范

> 测试基建已就绪（pytest + pytest-asyncio 已装，`backend/tests/` 已建）。

- **任何新功能必须附带测试；修 bug 先写复现该 bug 的失败测试**。
- 测试文件镜像被测模块：`app/services/apifox/case_service.py → tests/services/apifox/test_case_service.py`。
- 命名 `test_<行为>_<条件>_<期望>`；强制 AAA 结构；每个测试只断言一个行为。
- 同一行为多组输入用 `@pytest.mark.parametrize`，禁止复制粘贴。
- 用例设计先出清单（等价类 / 边界 / 异常路径）给评审确认再写实现：空值/零/负数/超长/Unicode；状态逻辑覆盖合法迁移 + 至少一组非法迁移。
- **禁止假测试**：仅断言不抛异常、断言 mock 被调但不验参数，都不算测试。
- 自查标准：删掉任一被测行为，至少一个测试要失败。
- 新代码行覆盖 ≥ 80%。

详见 `.claude/rules/testing.md`。

## 十四、Git 提交规范

| type | 说明 |
|------|------|

- 涉及 3 个以上文件的改动先出计划（Plan Mode），确认后再动手。

## 十五、注释与文档

- 注释只解释**「为什么」**，不复述**「是什么」**；不写大段解释性注释。
- 模块级 docstring 说明该文件职责与关键约束（正面范式见各 `apifox/*.py` 头部）：

```python
"""Apifox 接口用例 · 数据访问层（用例 + 断言/提取子表）。

不含业务校验与权限；不提交事务（service commit）。子表 bulk replace 由 service 编排。
"""
```

- 枚举类字段在列定义处用注释列出全部取值。
- TODO / FIXME 统一标记：`# TODO:` 待办、`# FIXME:` 待修复。
- 公开函数在类型注解已表达签名时无需赘述参数，仅在有非显然约束/副作用时补 docstring。
